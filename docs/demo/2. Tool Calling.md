上一块讲的是 LLM API 调用，也就是“怎么让模型回答问题”。

这一块讲 **Tool Calling（工具调用）**，也就是：**怎么让模型不只是回答，而是决定调用某个工具，并把工具结果继续用于下一步推理。**

这是 Agent 的核心能力之一。

- **普通聊天机器人**：
  
  ```text
  用户提问 ──> LLM 直接回答
  ```
- **带工具调用的 Agent**：
  
  ```text
  用户提问 ──> LLM 判断需要工具 ──> 后端执行工具 ──> 把工具结果交还给 LLM ──> LLM 再回答
  ```

例如用户问：“帮我看看这个 GitHub 仓库最近一次 commit 改了什么？”

1. **用户提问**
2. **LLM 判断**：需要调用 GitHub 工具
3. **调用**：`get_recent_commits(repo_url)`
4. **获取结果**：拿到 commit 信息
5. **推理与总结**：LLM 基于真实结果总结并回答

## 一、Tool Calling 到底是什么？

你可以把它理解成：

> **模型不会真的执行函数，它只是“提出要调用哪个函数，以及传什么参数”。真正执行函数的是你的后端代码。**

这一点非常重要。模型不是直接操作数据库、文件系统、浏览器、GitHub API。模型只是输出类似这样的意图 JSON：

```json
{
  "tool_name": "search_code",
  "arguments": {
    "query": "ModuleNotFoundError",
    "top_k": 5
  }
}
```

然后你的后端看到这个结果，才真正执行：

```python
search_code(query="ModuleNotFoundError", top_k=5)
```

执行完之后，再把结果塞回模型作为上下文：

> 工具 `search_code` 返回：
> 
> 1. `app/main.py` 里 `import repo_mind.core`
> 2. [pyproject.toml](file:///f:/VScode%20Workspace/Python_workspace/AgentLearning/pyproject.toml) 里没有配置 package path

最后模型基于这个真实的工具执行结果回答用户。

## 二、Tool Calling 的完整流程

你一定要记住这个循环：

```text
       User
        ↓
       LLM
        ↓
   Tool Call Request (工具名及参数)
        ↓
 Backend Executes Tool (后端执行工具)
        ↓
   Tool Result (工具执行结果)
        ↓
       LLM (进行下一步推理)
        ↓
   Final Answer (最终回答)
```

在 Agent 项目里，这个循环可能执行多次：

1. **用户问**：为什么项目测试失败？
2. **LLM 调用**：`read_file` 读取测试报错。
3. **LLM 调用**：`search_code` 搜索相关函数。
4. **LLM 调用**：`read_file` 查看实现。
5. **LLM 调用**：`run_tests` 复现错误。
6. **LLM 给出**：最终原因和修复建议。

这就已经接近 Claude Code / Codex 的工作方式了。

## 三、一个工具由什么组成？

一个工具通常包含三部分：

1. **工具名称 (Name)**
2. **工具描述 (Description)**
3. **参数模式 (Parameter Schema)**

比如你要做一个代码搜索工具：

```python
tool = {
    "name": "search_code",
    "description": "在代码仓库中搜索和用户问题相关的代码片段",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "搜索关键词或语义查询"
            },
            "top_k": {
                "type": "integer",
                "description": "返回最相关的代码片段数量"
            }
        },
        "required": ["query"]
    }
}
```

> [!IMPORTANT]
> **工具的 `description` 非常关键**，因为模型是根据工具描述来判断什么时候调用该工具的。
> 如果你的工具描述写得太模糊（如 `"搜索东西"`），模型可能乱用。
> 更好的描述是：`"当用户问题需要定位仓库中的函数、类、报错、配置项或调用链时，使用该工具搜索代码片段。"`

## 四、Tool Calling 和普通函数调用有什么区别？

- **普通后端函数调用**：在代码里写死逻辑：
  
  ```python
  if user_intent == "search":
      search_code(query)
  ```
- **Tool Calling**：由模型根据任务上下文**动态选择**工具和参数：
  - 这个问题是否需要 `search_code`？
  - `query` 应该是什么？
  - `top_k` 应该是多少？
  - 调用完之后下一步要不要继续调用 `read_file`？

这就是 Agent 的味道。

## 五、最小 Tool Calling 例子

假设你做一个天气助手。

1. 你定义一个工具函数：
   
   ```python
   def get_weather(city: str):
       return f"{city} 今天 26℃，多云"
   ```
2. 你把工具定义告知模型。
3. 用户问：“今天北京天气怎么样？”
4. 模型不会直接回答，而是返回：
   
   ```json
   {
     "tool_name": "get_weather",
     "arguments": {
       "city": "北京"
     }
   }
   ```
5. 后端执行：`result = get_weather("北京")`，得到 `"北京今天 26℃，多云"`。
6. 后端将工具结果作为消息追加入 messages 列表，再次调用模型。
7. 模型最终回答：“北京今天多云，气温约 26℃，适合正常出行。”

真实天气数据不是模型生成的，而是工具返回的。**LLM 负责判断和表达，工具负责获取真实信息或执行真实动作。**

## 六、Agent 项目里常见的工具类型

对于你想做的智能体项目，工具一般分这几类：

### 1. 检索类工具

- 例如：`search_code`、`search_docs`、`search_web`、`search_vector_db`。
- **作用**：帮模型找信息。在代码理解智能体里，最核心的工具就是搜索仓库代码、读取文件、查找调用关系。

### 2. 读取类工具

- 例如：`read_file`、`read_directory`、`get_file_tree`、`get_commit_diff`。
- **作用**：让模型看到具体文件内容。代码类 Agent 不能只靠向量检索片段，很多时候需要读取完整文件。

### 3. 执行类工具

- 例如：`run_tests`、`run_linter`、`run_command`、`execute_python`。
- **作用**：让 Agent 不只是分析，还能通过测试和编译反馈进行验证。

### 4. 写入类工具

- 例如：`write_file`、`apply_patch`、`create_pr`。
- **作用**：让 Agent 能够自动修改文件。这类工具风险很高，必须加**权限控制**和**用户人工确认**，不能让模型随便改文件。

### 5. 外部 API 工具

- 例如：GitHub API、Jira API、飞书 API、数据库查询等。
- **作用**：让 Agent 接入真实业务系统，这也是体现工程落地能力的重要指标。

## 七、Tool Calling 的关键不是“能调用”，而是“能控住”

很多人做 demo 只做到“模型可以调用工具”，但工程里真正难的是：

- 模型什么时候该调用？
- 参数是否合法？
- 工具执行失败怎么办？
- 是否允许这个工具被调用？
- 调用次数会不会无限循环？
- 工具结果太长怎么办？
- 模型会不会编造工具结果？

---

## 八、Tool Calling 常见问题与优化方案

### 问题 1：模型乱调用工具

- **场景**：用户只是问：“你是谁？”，模型却调用了 `search_code`。
- **优化**：工具描述要精准，并在 `system prompt` 里加上约束：*“只有当回答问题必须依赖仓库信息时，才调用 search_code。如果可以直接回答，不要调用工具。”*

### 问题 2：模型传错参数

- **场景**：工具定义需要参数 `{"file_path": "..."}`，模型却传了 `{"path": "..."}`。
- **优化**：使用严格的 Pydantic 或 JSON Schema 校验参数，参数不合法时返回错误信息，让模型重新生成。

### 问题 3：工具返回太长

- **场景**：`read_file` 读取了一个 5000 行文件，直接塞回模型会导致爆上下文。
- **优化**：限制单次读取行数，参数中支持 `start_line` / `end_line`；或者先返回摘要和结构。
  
  ```json
  {
    "file_path": "src/main.py",
    "start_line": 1,
    "end_line": 120
  }
  ```

### 问题 4：无限循环调用工具

- **场景**：LLM 调用 `search_code` -> 结果不满意 -> 继续 `search_code` -> 陷入死循环。
- **优化**：设置最大调用轮数限制（例如 `MAX_TOOL_STEPS = 5`），记录调用历史，相同参数不允许重复调用。

### 问题 5：危险工具安全控制

- **场景**：模型想要执行类似 `rm -rf /` 的 shell 命令或清空数据库。
- **优化**：加白名单、加沙箱环境、限制只读权限。对于写入操作，必须由用户确认。
  
  > **面试亮点**：“我把读取类工具和写入类工具分了权限等级，默认只允许只读工具自动调用，写入和执行命令需要用户确认或在沙箱内运行。”

## 九、在 RepoMind 这种项目里，Tool Calling 应该怎么设计？

如果你的项目是代码仓库理解 Agent，最小工具集（P0 级）建议设计如下：

### 1. `get_project_tree`

- **用途**：查看项目目录结构，了解项目是前端、后端、还是库，寻找关键文件。
- **参数**：`{"max_depth": 3}`
- **返回**：
  
  ```text
  .
  ├── src/
  │   ├── agent/
  │   ├── retriever/
  │   └── server.py
  ├── tests/
  ├── pyproject.toml
  └── README.md
  ```

### 2. `search_code`

- **用途**：根据关键词或语义在仓库中搜索相关的代码片段。
- **参数**：`{"query": "ModuleNotFoundError import path", "top_k": 5}`
- **返回**：匹配的代码片段及文件位置。

### 3. `read_file`

- **用途**：读取指定文件的指定行范围，获取完整代码上下文。
- **参数**：`{"file_path": "src/repomind/loader.py", "start_line": 1, "end_line": 120}`
- **返回**：行范围内的代码内容。

### 4. `run_tests` (P1 级)

- **用途**：运行测试以验证问题是否复现或被修复。
- **参数**：`{"test_path": "tests/test_loader.py"}`
- **返回**：测试是否通过、报错日志及耗时。

## 十、Tool Calling 在面试里怎么讲？

> [!TIP]
> **面试回答话术**：
> “我的 Agent 不是直接把用户问题发给模型，而是给模型注册了一组工具，比如项目结构读取、代码搜索、文件读取和测试运行。模型会根据用户问题决定是否调用工具，并生成结构化的工具参数。后端负责校验参数、执行工具、记录日志，再把工具结果作为上下文返回给模型，让模型基于真实代码证据回答。”

如果面试官追问：“模型真的会直接运行函数吗？”
你可以回答：“严格来说不会。模型只是输出一个包含工具名和参数的 JSON。真正执行函数的是我们的后端代码，后端拿到结果后再包装成 messages 发送给模型继续推理。”

## 十一、你现在需要掌握到什么程度？

不要一上来搞复杂的多工具 Agent。你先做到这几个就够：

1. 能定义一个工具 Schema。
2. 能让模型根据用户问题选择工具。
3. 能解析模型返回的 `tool_calls` 结构。
4. 能在后端匹配并执行真实 Python 函数。
5. 能把工具执行结果重新拼装传给模型。
6. 能限制最大工具调用次数和处理工具执行失败。

## 十二、最小练习任务

你可以写一个非常小的 demo 项目，用户输入报错日志，Agent 可调用以下工具：

- `search_code(query)`
- `read_file(file_path, start_line, end_line)`

流程：

1. **用户**：“为什么出现 `ModuleNotFoundError`？”
2. **模型调用**：`search_code("ModuleNotFoundError import")`
3. **后端返回**：返回 `setup.py` / `pyproject.toml`
4. **模型调用**：`read_file("pyproject.toml")`
5. **后端返回**：返回包配置内容
6. **模型分析**：得出包路径没有声明的结论，输出修复建议。

## 十三、这一块你最该记住的一句话

> **Tool Calling 的本质是：LLM 负责决策和参数生成，后端负责真实执行和安全控制。**

真正的 Agent 不是仅仅有工具，而是能根据任务**动态选择工具**、**使用工具结果继续推理**并**能多步自我修正与处理失败**。
