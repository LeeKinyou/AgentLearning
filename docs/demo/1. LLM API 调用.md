你可以先把 LLM API 理解成一个特殊的后端接口：
- **普通后端接口**：输入结构化参数，返回确定结果。
- **LLM 接口**：输入上下文、指令、历史消息，返回一段概率生成的文本或结构化结果。

Agent、RAG、Tool Calling、多智能体，本质上都是围绕 LLM API 做的一层工程封装。所以你必须先搞懂：**怎么稳定、可控、低成本地调用模型**。

---

## 一、LLM API 最核心的概念：messages

现在主流大模型接口基本都不是简单传一个字符串，而是传一个消息列表。

```python
messages = [
    {
        "role": "system",
        "content": "你是一个专业的代码审查助手，只能基于用户提供的代码回答。"
    },
    {
        "role": "user",
        "content": "帮我看看这个函数有什么问题：..."
    }
]
```

你要重点理解三个角色：

### 1. system
这是最高层指令，用来规定模型身份、边界、输出格式、禁止行为。
例如：
- 你是一个代码审查助手。
- 你必须指出 bug、风险和修改建议。
- 不要编造不存在的代码。
- 输出必须包含：`问题`、`原因`、`修改建议`。

在项目里，`system prompt` 通常是固定模板。

### 2. user
用户输入的问题。例如：
> 为什么我的 Python 项目启动时报 `ModuleNotFoundError`？

这个内容是动态的，每次用户请求都不一样。

### 3. assistant
模型之前回复过的内容。多轮对话时，需要把历史 `assistant` 消息也传进去，否则模型不知道前面聊了什么。

```python
messages = [
    {"role": "system", "content": "你是一个 Agent 开发助手"},
    {"role": "user", "content": "什么是 RAG？"},
    {"role": "assistant", "content": "RAG 是检索增强生成..."},
    {"role": "user", "content": "那它和 Agent 有什么关系？"}
]
```

这就是所谓的 **上下文窗口**。你现在要记住一句话：
> **LLM 没有真正的“记忆”，它每次都只看你这次请求传进去的 messages。**

所以 Agent 工程里的 `memory`、`history`、`state`，本质都是你在后端帮模型维护上下文。

## 二、一次 LLM 调用到底发生了什么？

整个调用流程可以表示为：

```text
用户问题
  ↓
后端组装 messages
  ↓
调用 LLM API
  ↓
模型生成回复
  ↓
后端解析结果
  ↓
返回给用户 / 继续执行下一步
```

例如一个代码问答 Agent：
1. **用户问**：这个报错怎么解决？
2. **系统检索**：检索相关代码。
3. **组装**：把“用户问题 + 相关代码 + 输出要求”组装成 `prompt`。
4. **调用**：调用模型。
5. **返回**：模型返回原因和修复建议。

所以你做智能体项目时，不是简单地“把问题丢给模型”，而是要设计：
- 我给模型什么上下文？
- 我让模型扮演什么角色？
- 我要求模型按什么格式输出？
- 我如何验证模型输出是否可用？

---

## 三、几个你必须掌握的参数

### 1. model
选择调用哪个模型。例如：
- `model="gpt-4"`
- `model="claude-3-5-sonnet"`
- 本地模型：`model="qwen"`

面试时不用纠结具体模型名字，重点是知道不同模型的差异：
- **强模型**：效果好，贵，慢。
- **小模型**：便宜，快，但推理和代码能力弱一些。

实际项目里一般会根据任务选模型：
- **复杂分析**：强模型
- **简单分类**：小模型
- **代码修复**：代码能力强的模型
- **批量处理**：便宜模型

### 2. temperature
控制模型输出随机性（取值范围一般在 `0` 到 `2` 之间）。
- `temperature=0`：更稳定，更适合代码、结构化输出、分类、工具调用。
- `temperature=0.7`：更发散，更适合创意写作、方案 brainstorm。

对于 Agent 开发，通常建议设置较小的值（如 `0` 或 `0.2`），因为 Agent 需要稳定，不能每次都胡乱发挥。

### 3. max_tokens
限制模型最多输出多少 token。这影响：
- 输出长度
- 调用成本
- 响应速度
- 是否被截断

比如你让模型生成很长报告，但 `max_tokens` 设置太小，就会输出到一半断掉。

### 4. stream
是否流式输出。
- **普通输出**：等模型全部生成完 → 一次性返回。
- **流式输出**：模型生成一点 → 后端推一点 → 前端显示一点（例如 ChatGPT、Claude Code 这种边生成边显示，就是 streaming）。

在项目里，streaming 主要改善用户体验。尤其是代码分析、长报告、RAG 问答，用户不想干等。

## 四、最小 LLM 调用代码长什么样？

下面是使用标准 Python SDK 调用 LLM 的示例代码：

```python
def call_llm(user_question: str):
    messages = [
        {
            "role": "system",
            "content": "你是一个专业的后端开发助手，回答要准确、简洁、可执行。"
        },
        {
            "role": "user",
            "content": user_question
        }
    ]

    response = client.chat.completions.create(
        model="your-model-name",
        messages=messages,
        temperature=0.2,
        max_tokens=1000,
    )

    return response.choices[0].message.content
```

你不需要死记 API 名字，因为不同厂商 SDK 不一样。你要掌握的是这个核心数据流向结构：
```text
client ──> model ──> messages ──> parameters ──> response ──> parse content
```

> [!TIP]
> **面试加分表达**：
> “我一般会在后端封装一个 `LLMClient`，统一处理模型调用。输入包括 `system prompt`、用户问题、上下文信息和参数配置。调用时会控制 `temperature`、`max_tokens`，并根据任务决定是否使用流式输出。返回结果后，我会做结构化解析、异常捕获、日志记录和重试，避免业务逻辑直接耦合具体模型厂商。”

## 五、Prompt 不是随便写一句话

很多初学者以为 prompt 是这样：
> 请帮我分析这个项目。

这太弱了。工程里的 prompt 应该有严密的结构。例如代码审查 prompt 可以这样设计：

```text
你是一个资深后端代码审查助手。

【任务】
根据用户提供的代码和问题，找出潜在 bug、设计问题和改进建议。

【约束】
1. 只能基于提供的代码回答。
2. 不要编造不存在的文件、函数或接口。
3. 如果信息不足，明确说明缺少什么。
4. 输出要具体，不要泛泛而谈。

【输出格式】
- 问题概述
- 可能原因
- 证据代码
- 修改建议
- 风险提醒
```

这个就比一句“帮我分析”稳定得多。所以你要记住：**Prompt 工程不是写作文，而是给模型写接口协议。**你越明确，模型越稳定。

## 六、为什么要结构化输出？

比如你让模型返回：“这个代码有三个问题，第一...”，这种纯文本格式人能看懂，但程序不好处理。
如果你要让后端继续执行下一步，就最好让模型返回 JSON。例如：

```json
{
  "summary": "数据库连接未正确关闭",
  "risk_level": "high",
  "issues": [
    {
      "file": "db.py",
      "line": 42,
      "problem": "连接对象没有释放",
      "suggestion": "使用上下文管理器管理连接"
    }
  ]
}
```

这样后端可以继续做：
- 展示风险等级
- 定位文件行号
- 生成修复 patch
- 写入报告
- 触发测试

这就是 Agent 和普通聊天机器人的区别：**普通聊天机器人只要回答人能看懂就行，而 Agent 需要输出能被程序消费的结构化结果。**

## 七、LLM API 调用最常见的坑

- **坑 1：没有 system prompt**
  只传用户问题，模型容易乱答。
- **坑 2：上下文塞太多**
  把整个项目代码全部塞进去，会导致成本高、速度慢、超过上下文限制、模型抓不住重点。所以后面才需要 RAG。
- **坑 3：没有错误处理**
  真实项目里 API 调用可能因为网络错误、限流、超时、余额不足、JSON 解析失败等原因报错。所以在后端必须进行封装：
  ```python
  try:
      result = call_llm(messages)
  except TimeoutError:
      retry()
  except RateLimitError:
      wait_and_retry()
  except JSONDecodeError:
      ask_model_to_fix_json()
  ```
- **坑 4：没有日志**
  Agent 出问题很难排查，因为模型是黑盒。你必须记录用户输入、检索到的上下文、最终 prompt、模型输出、工具调用结果、耗时、token 消耗及错误信息，才能快速定位问题是出在检索、prompt 还是模型理解。

## 八、你现在应该掌握到什么程度？

不要追求把所有厂商 API 都背下来。你先达到这个程度就够投实习：
1. 能用 Python 或 Java 调通一个 LLM API。
2. 知道 messages 的 role：`system` / `user` / `assistant`。
3. 知道 `temperature`、`max_tokens`、`streaming` 的作用。
4. 能写一个结构化 prompt。
5. 能让模型返回 JSON 并使用 Pydantic 解析。
6. 能做基本异常处理和日志记录。

## 九、给你一个最小练习任务

你可以做一个非常小的 demo，输入一段 Python 报错日志，要求模型输出如下结构化 JSON：
```json
{
  "error_type": "...",
  "possible_reason": "...",
  "suggestion": "...",
  "need_more_context": true
}
```
这个练习能帮你打通 LLM API 调用、system prompt、结构化输出、JSON 解析及异常处理的核心流程。

## 十、这一块和 Agent 的关系

```text
LLM API 调用 ──> 大脑接口
Tool Calling  ──> 大脑决定调用工具
RAG           ──> 给大脑补充资料
Agent 工作流   ──> 让大脑多步思考和执行
工程后端化     ──> 让系统稳定运行
项目表达       ──> 让面试官相信你真的做过
```
所以第一块不是最炫的，但它是不可或缺的地基。
