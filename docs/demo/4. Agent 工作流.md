前面三块是：
- **LLM API 调用**：让模型能回答。
- **Tool Calling**：让模型能调用工具。
- **RAG**：让模型基于资料回答。

今天讲第 4 块：**Agent 工作流**。

你可以先记住一句话：
> **Agent 工作流 = 让模型围绕一个目标，进行多步决策、调用工具、观察结果、修正计划，直到完成任务或达到停止条件。**

- **普通 LLM 调用**（一次性）：
  ```text
  用户问题 ──> 模型回答
  ```
- **Agent 工作流**（循环式）：
  ```text
  用户目标 ──> 计划 (Plan) ──> 执行 (Act) ──> 观察结果 (Observe) ──> 修正 (Revise) ──> 继续执行/停止
  ```

这就是常说的：**Plan ──> Act ──> Observe ──> Revise** 循环。

---

## 一、为什么需要 Agent 工作流？

因为很多任务（尤其是复杂的工程任务）不是一次模型调用就能完成的。
例如用户问：“为什么我的项目启动时报 `ModuleNotFoundError`？”
- **普通 LLM**：最多只能根据先验知识猜测可能是包没装、路径不对、环境问题。
- **Agent**：应该多步收集证据：
  1. 查看项目结构
  2. 搜索报错关键词
  3. 读取 `pyproject.toml`
  4. 读取入口文件并分析包路径配置
  5. 给出带代码证据的准确修改建议

Agent 不应该一上来就回答，而应该先收集充足的证据。这就是工作流的价值。

---

## 二、Agent 的基本循环

最经典的 Agent 循环是：
```text
用户目标 ──> LLM 思考下一步 ──> 选择工具 ──> 后端执行工具 ──> 得到观察结果 (Observation) ──> LLM 基于结果继续判断 ──> 直到满足停止条件
```

可以用如下伪代码表示：

```python
def run_agent(user_task):
    state = {
        "task": user_task,
        "steps": [],
        "evidence": [],
    }

    for i in range(MAX_STEPS):
        action = llm_decide_next_action(state)

        if action.type == "final_answer":
            return action.content

        tool_result = execute_tool(action.tool_name, action.arguments)

        state["steps"].append(action)
        state["evidence"].append(tool_result)

    return summarize_with_limited_evidence(state)
```

这里引入了一个非常核心的概念：**State（状态）**。

---

## 三`、State`：Agent 的状态管理

Agent 在循环中需要记住：
- 用户原始问题是什么
- 已经调用过哪些工具、传了什么参数
- 每个工具返回了什么结果
- 目前已经拿到了哪些关键证据
- 当前任务进展以及还有哪些信息缺失
- 是否已经达到了回答的条件

这些信息合起来就是 Agent 的 `state`。例如在 RepoMind 里，一个 `state` 对象可以长这样：

```json
{
  "question": "为什么启动时报 ModuleNotFoundError: No module named 'repomind'？",
  "current_plan": [
    "查看项目结构",
    "读取 pyproject.toml",
    "搜索 repomind import",
    "分析包路径配置"
  ],
  "tool_history": [
    {
      "tool": "get_project_tree",
      "args": {"max_depth": 3},
      "result": "src/repomind, tests, pyproject.toml..."
    },
    {
      "tool": "read_file",
      "args": {"file_path": "pyproject.toml"},
      "result": "..."
    }
  ],
  "evidence": [
    {
      "file": "pyproject.toml",
      "lines": "1-40",
      "content": "..."
    }
  ],
  "missing_info": [
    "还没有查看入口文件"
  ]
}
```

普通函数是简单的输入与输出，而 **Agent 则是一个随着步骤不断变化的任务状态机**。

---

## 四、Plan：计划阶段

计划就是让模型先判断：
- 这个任务需要分几步完成？
- 需要用到哪些工具？
- 先查什么，后查什么？

### 1. 计划不要太长
新手 Agent 容易一上来就生成 20 步计划，这在实际工程里是无法落地的。推荐做法是：
- 先生成 3-5 步的**粗计划**。
- 每一步执行后根据实际返回动态调整计划，因为 Agent 一开始不可能拥有全部信息。

### 2. 计划要和工具能力匹配
如果你的工具集只有 `search_code`、`read_file`、`get_project_tree`，就必须在 `system prompt` 里限制模型不能计划去执行“启动 Docker 容器”或“自动创建 PR”等不存在的操作。

### 3. 计划要有清晰的停止条件
- 收集到足够证据时，输出最终答案。
- 达到最大步数上限时，基于已有证据回答。

---

## 五、Act：执行阶段

模型根据当前的 `state` 决定下一步要调用的工具和参数。

```json
{
  "tool": "get_project_tree",
  "arguments": {
    "max_depth": 3
  },
  "reason": "需要先了解项目是否采用 src layout，以及包目录是否存在"
}
```

后端收到这个 action 请求后，真正执行工具并将结果写回 `state`。

> [!IMPORTANT]
> **模型只决策，后端负责执行与控制**。
> 后端接收到模型决策后，必须进行白名单校验、参数检查、目录安全限制、捕获执行异常，再把结果返回给 Agent state。例如模型如果要求执行 `{"tool": "run_shell", "arguments": {"command": "rm -rf /"}}`，后端必须直接拦截。

---

## 六、Observe：观察阶段

把工具执行的真实结果作为 `Observation` 返回给模型。
例如工具返回了项目目录结构，模型观察后得出：“这是典型的 `src layout` 项目，下一步应该查看 `pyproject.toml` 中的包路径配置。”
模型便据此发起下一步调用。**工具的返回结果不是最终答案，而是下一步推理的证据。**

---

## 七、Revise：修正阶段

模型根据新获得的证据随时调整原先的计划。
例如：一上来的计划是查看 `pyproject.toml`，但执行 `get_project_tree` 后发现项目根本没有 `pyproject.toml`，只有 `setup.py`。
这时候模型需要有修正能力，将下一步计划自动改为“查看 `setup.py` 并检查 `packages` 配置”。
**没有 Revise，Agent 只是一个死板的脚本；有了 Revise，它才具备真正的智能。**

---

## 八、什么时候继续，什么时候停止？

这是 Agent 控制中最核心的问题，要防止 Agent 无限循环调用工具。

### 1. 可以停止的情况
- **证据已充分**：已拿到了确定性的配置和代码，足以解答用户问题。
- **达到步数上限**：例如设置了 `MAX_STEPS = 6`，达到上限仍未完全确认，也必须停止，让模型基于已有证据进行最合理的推测并指出缺失的确认信息。
- **缺少必要输入**：用户没有提供报错日志或仓库路径，模型无法继续。
- **触发高风险操作**：如需要修改代码、提交 PR 或执行 shell 命令，需要停止并请求用户人工确认。

### 2. 应当继续的情况
- **只有结论，没有代码/配置证据**：要求模型必须调用工具拿到证据。
- **证据存在冲突**：例如 README 里写是用 Poetry 启动，但项目中只有 `requirements.txt`。此时需要进一步查看 lock 文件或 CI 配置。
- **有函数调用，但找不到定义**：必须继续追溯函数的具体实现。
- **工具调用临时失败**：例如读取文件报错，应尝试调整路径重新读取，而不是直接崩溃退出。

## 九、Agent 工作流的常见模式

### 1. ReAct 模式 (Reasoning + Acting)
```text
Thought: 我需要先查看项目结构
Action: get_project_tree
Observation: 项目采用 src layout
Thought: 需要查看 pyproject.toml
Action: read_file
Observation: pyproject.toml 没有 package_dir
Thought: 证据已充分，可以得出结论
Final Answer: 原因是...
```
这是最经典的 Agent 模式。在工程里，`Thought` 过程通常记录在日志中，不直接展示给用户。

### 2. Plan-and-Execute 模式
先生成整体规划，再按步骤执行。适合长任务和复杂开发任务（如“给项目增加登录功能”），执行中需要配合 Revise 机制动态调整。

### 3. Router 模式
先用模型对任务分类，再路由到不同的短链路或专用工具链（如代码解释走简单检索；Bug 定位走完整 ReAct 循环）。这在生产环境里极其有效，能极大地降低 token 消耗和响应时延。

### 4. 状态图模式 (State Machine / Graph)
使用类似 LangGraph 的框架，不再让模型完全自由发挥，而是通过代码把流程限制在图节点和条件跳转中。
```text
parse_question ──> retrieve_context ──> decide_next_action ──> execute_tool ──> evaluate ──> final_answer
```
这种模式最适合生产环境，因为它极大地提升了 Agent 行为的确定性和可控性。

## 十、RepoMind 的工作流设计示例

1. **`classify_task`**：判断任务类型。
2. **`build_initial_query`**：提取报错关键词、函数名、文件名。
3. **`retrieve_context`**：进行 Hybrid search 检索。
4. **`inspect_evidence`**：评估证据是否足够。若不足，调用 `read_file` 等工具补充。
5. **`analyze`**：基于多步收集的证据分析根因。
6. **`final_report`**：输出包含结论、证据文件行号、可能原因及修复建议的报告。

---

## 十一、工作流里最重要的工程安全与控制

- **最大步数限制**：`MAX_STEPS = 6`。
- **工具白名单**：不同任务类型限制不同的工具访问权限。
- **参数校验**：严格验证文件路径是否安全（防止类似 `../../etc/passwd` 的路径穿越攻击）。
- **工具返回内容截断**：例如限制读取文件工具单次最大返回 200 行，避免爆上下文。
- **调用去重**：如果模型连续使用相同参数重复调用同一工具，后端直接予以拦截并报错反馈，强迫其更换策略。
- **结构化日志**：记录每一步的状态、action、参数、返回、延迟和 token 消耗，这是调试的生命线。

## 十二、Agent 工作流面试怎么讲？

> [!TIP]
> **面试回答话术**：
> “我的 Agent 采用类似 ReAct 的状态循环，在后端维护一个包含任务状态、工具历史和证据列表的 state。每一步由模型根据当前 state 决定调用哪个工具，后端执行后把 observation 写回 state，再由模型判定是继续收集证据还是生成最终报告。在工程上，我通过设置最大步数限制、参数路径校验、工具白名单以及重复调用拦截等机制，确保工作流的稳定与安全。”
