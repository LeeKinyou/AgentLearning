前两块讲的是：

- **LLM API 调用**：怎么让模型回答。
- **Tool Calling**：怎么让模型调用工具。

今天讲第 3 块：**RAG（检索增强生成）**。这是智能体开发里最常见、最重要、最容易被问到的基础能力之一。

你可以先记住一句话：

> **RAG = Retrieval-Augmented Generation，检索增强生成。也就是先从知识库/代码库/文档里检索相关资料，再把资料交给大模型回答。**

简单说就是：

- **不是直接问模型**：“这个项目的 `QueryEngine` 函数有什么问题？”（模型没有代码上下文，只能乱猜或拒绝）。
- **而是先检索项目代码**：找到 `QueryEngine` 函数、相关调用以及配置文件。
- **再把检索结果交给模型**：“请基于以下代码证据进行回答。”

---

## 一、为什么需要 RAG？

因为大模型有几个天然问题：

1. **不知道你的私有数据**：没有你的项目源码。
2. **训练知识可能过期**：无法掌握最新发布的技术或 API。
3. **幻觉问题**：容易编造不存在的信息。
4. **上下文窗口限制**：不能把几万行、几十万行的整个项目直接塞给模型。
5. **注意力分散**：即使塞进去了，长上下文中模型也容易抓不住重点。

RAG 的核心思路是：**不把所有资料都给模型，只把最相关的一小部分给模型。**

```text
用户问题
  ↓
检索相关文档 / 代码片段 (Retrieve)
  ↓
把相关片段塞进 Prompt (Augment)
  ↓
模型基于片段回答 (Generate)
```

---

## 二、RAG 的完整流程

标准 RAG 流程可以拆分为两个阶段：

### 阶段 1：离线构建知识库

提前把资料处理好并持久化：

```text
文档/代码/PDF ──> 加载 (Loading) ──> 切块 (Chunking) ──> 向量化 (Embedding) ──> 存入向量库 (Vector Store)
```

### 阶段 2：在线检索回答

用户提问时实时走以下流程：

```text
用户问题 ──> 问题向量化 (Query Embedding) ──> 相似度检索 (Vector Retrieval) ──> 重排序 (Rerank) ──> 组装 Prompt ──> LLM 生成回答
```

> [!TIP]
> **面试加分表达**：
> “我先对文档进行加载和切块，然后用 embedding 模型把 chunk 转成向量，存入向量库。用户提问时，我把 query 也转成向量，在向量库里做相似度检索，取 top-k 片段作为上下文传给 LLM。必要时会用 reranker 对候选片段重排序，最后要求模型基于检索证据回答，并返回引用来源。”

---

## 三、Document Loading：文档加载

RAG 第一步是把资料读进来。在你的场景里，最重要的是：**代码文件、README、配置文件、测试文件和报错日志**。

对于 RepoMind 这种项目，文档加载器（Document Loader）需要做的事情是：

- 遍历仓库目录并过滤无用文件（如 `.git`、`.venv`、`__pycache__`）。
- 读取代码和文档内容。
- 保留元数据（Metadata）。

元数据非常重要，一个 chunk 应该包含：

```json
{
  "content": "def search_code(...): ...",
  "metadata": {
    "file_path": "src/repomind/search.py",
    "start_line": 20,
    "end_line": 58,
    "language": "python"
  }
}
```

如果没有 `metadata` 中的文件路径和行号，回答最后就无法附带真实的证据来源，模型只能泛泛而谈。

---

## 四、Chunking：切块

切块是 RAG 里最容易被低估的部分。它是指把长文档拆成小片段（例如一个 3000 行的文件不可能整体塞给模型）。

### 1. 为什么需要 overlap（重叠度）？

如果你完全不重叠（如 `chunk 1` 是 1-100 行，`chunk 2` 是 101-200 行），那么跨边界的函数（如定义在 95 行，核心实现在 105 行）就会被切断，检索时就拿不到完整的上下文。
所以一般会加重叠部分（如 `chunk 1` 是 1-100 行，`chunk 2` 是 80-180 行）。

### 2. Chunk Size 怎么选？

- **普通文档**：500-1000 tokens 一个 chunk。
- **代码文件**：最好不要机械地按字符数切，而是**按函数、按类、按逻辑块**切（可以使用 AST 分析来做代码切块），保留 `import` 关系和行号。
- **Markdown**：按标题层级切。
- **日志**：按一次错误堆栈切。

### 3. 切块太大和太小的问题

- **Chunk 太大**：检索不精准、token 成本高、响应慢，模型抓不住重点。
- **Chunk 太小**：检索精准，但上下文断裂（如只检索到 `return result`，模型根本不知道 `result` 是什么）。
  
  > **核心原则**：Chunk 要小到便于检索，大到足够表达完整语义。

---

## 五、Embedding：向量化

Embedding 是把文本转换成一串数字向量，用来表达文本的**语义**。
例如：“数据库连接失败”、“connection refused” 和 “无法连接 MySQL” 字面完全不同，但语义接近，embedding 会把它们映射到向量空间中比较近的位置。这样即使没有精确匹配的字词，系统也能检索到相关的代码或配置。

### 向量搜索和关键词搜索的区别

- **关键词搜索**：字面完全匹配。如搜索 `"UserService"` 或 `"ModuleNotFoundError"` 时，精准匹配非常有效。
- **向量搜索**：语义匹配。如搜索 `"登录失败"` 时，能匹配到包含 `"auth error"` 或 `"unauthorized"` 的片段。

在真实工程里，通常采用 **混合检索（Hybrid Search）**：

```text
混合检索 = 关键词检索 (BM25) + 向量检索 (Embedding) + 规则过滤
```

> [!NOTE]
> 对于代码仓库，精确符号（函数名、类名、配置项）的检索至关重要。因此我们在 RepoMind 中采用混合检索，既能覆盖模糊语义，又能精准匹配特定符号。

---

## 六、Vector Store：向量数据库

向量数据库是存储 embedding 和元数据并进行相似度检索的地方。
常见向量库有：

- **FAISS / Chroma**：轻量，适合本地 demo。
- **pgvector**：基于 PostgreSQL 扩展，非常适合后端工程化。
- **Milvus / Qdrant**：适合更专业的大规模向量检索服务。

项目初期使用 `FAISS` 或 `Chroma` 即可满足本地开发和求职演示。

---

## 七、Top-k：检索多少条？

Top-k 就是指相似度最高的前 $k$ 个片段。

- **Top-k 太小**：容易漏掉关键信息。
- **Top-k 太大**：上下文太长，噪声变多，成本变高，模型容易被干扰。

代码 RAG 的推荐实践：

1. **第一步**：`top_k = 8` 或 `10`，先召回更多候选片段。
2. **第二步**：使用 **Rerank** 重排序，筛选出最相关的 3-5 个。
3. **第三步**：如果仍不够，让 Agent 动态调用 `read_file` 工具读取完整上下文。

---

## 八、Rerank：重排序

很多 demo 拿向量检索的 top-k 直接塞给 LLM，效果经常不稳定，因为向量检索返回的第一名不一定是最佳答案。
Rerank 的作用是：**使用交叉编码器（Cross-Encoder）等更精细的模型对初步召回的 top-k 候选片段重新打分，精选出最相关的结果。**

```text
向量检索 (召回 Top 20) ──> Rerank 重新打分 ──> 取 Top 5 塞入 Prompt
```

Embedding 检索负责“广撒网”，Rerank 负责“精筛选”。

---

## 九、Prompt 组装：把检索结果交给模型

检索到资料后，需要以清晰的结构填入 Prompt。例如：

```text
你是一个代码仓库分析助手。
请只基于给定的代码片段回答，不要编造任何信息。
如果证据不足，请说明还需要哪些文件。

【用户问题】
{question}

【检索到的代码证据】
[片段 1]
文件：src/search.py
行号：10-45
内容：
...

[片段 2]
文件：pyproject.toml
行号：1-30
内容：
...

【输出要求】
1. 结论
2. 证据
3. 可能原因
4. 修改建议
```

## 十、引用证据：让回答可验证

RAG 的核心价值是**回答必须能追溯到资料来源**。
例如，不要只说“问题出在配置文件没有声明包路径”，更好的回答是：“问题可能出在 `pyproject.toml` 没有正确配置包包含规则。证据是 `pyproject.toml` 第 12-20 行只声明了依赖，未声明 include 规则。”

证据引用要包含：**文件路径、行号、函数/类名**。这能大大提升系统的工程可信度。

## 十一、RAG 常见失败原因与解决方法

1. **检索不到（召回率低）**：用户问题与代码表达不一致。
   - *解决*：使用 embedding 语义检索、query rewrite（问题改写）、混合检索。
2. **检索到了但不完整**：只找到了函数调用，没找到函数定义。
   - *解决*：根据调用链追溯，或让 Agent 动态调用工具（如 `read_file`、`find_definition`）扩展上下文。
3. **检索结果太多太乱（噪声大）**：
   - *解决*：引入 Rerank、加上元数据过滤（Metadata Filtering，如按文件类型、目录过滤）。
4. **代码切块不合理**：函数被生硬切断。
   - *解决*：使用基于语法树（AST）的 Chunking，确保函数与类定义的完整性。
5. **模型不遵守约束**：拿了检索结果却开始自行幻觉发挥。
   - *解决*：严格 Prompt 约束，要求结论必须注明来源，无证据时输出“无法确认”。

---

## 十二、代码 RAG 和普通文档 RAG 的区别

| 特性       | 普通文档 RAG | 代码仓库 RAG              |
|:-------- |:-------- |:--------------------- |
| **内容类型** | 自然语言段落   | 结构化代码、配置、脚本           |
| **核心关系** | 文本上下文    | 依赖关系、调用链、继承、运行时逻辑     |
| **检索要求** | 语义相似度为主  | 精确符号（函数/类名）、路径匹配 + 语义 |
| **交互逻辑** | 一问一答     | 多步收集（结合文件读取、执行测试等）    |

所以代码 RAG 很难单靠一次向量检索解决，通常需要**结合静态分析、调用图、文件读取工具和测试运行工具**，让 Agent 多步收集证据。

---

## 十三、在 RepoMind 里，RAG 应该怎么做？

你可以把 RepoMind 的 RAG 设计为以下流程：

```text
用户输入问题/报错
      ↓
Query Rewrite (改写成便于检索的查询)
      ↓
Hybrid Retrieval (关键词 + 向量检索)
      ↓
Rerank (重排序最相关片段)
      ↓
Context Expansion (补充类定义、调用方等上下文)
      ↓
LLM Analysis (基于证据分析)
      ↓
Answer with Citations (输出带引用的结论与建议)
```

例如，用户输入 `ModuleNotFoundError: No module named 'repomind'`，系统先召回 `pyproject.toml`、`src/repomind/__init__.py` 以及安装说明，模型基于这些证据明确指出 packages 配置问题，给出执行 `pip install -e .` 的建议。

## 十四、RAG 面试怎么讲？

> [!TIP]
> **面试回答话术**：
> “我项目里的 RAG 流程分成离线索引和在线检索两部分。离线阶段会加载仓库代码和文档，过滤无关文件，然后按函数、类或 Markdown 标题进行切块，保留文件路径和行号等 metadata，再用 embedding 模型向量化并写入向量库。在线阶段，用户提问后会先进行 query rewrite，然后使用向量检索和关键词检索召回候选 chunk，再通过 rerank 选出最相关片段，最后把片段和引用信息一起传给 LLM，让模型基于证据回答。”

如果面试官追问：

- **“为什么不用直接把整个项目塞给模型？”**
  - 回答：“整个项目通常超过上下文窗口，而且成本高、速度慢、噪声大。RAG 只检索相关片段，减少上下文长度，并提供准确的引用来源，使回答可验证。”
- **“代码 RAG 和普通 RAG 有什么区别？”**
  - 回答：“普通 RAG 处理自然语言段落，代码 RAG 需要处理调用关系、继承、依赖和配置。因此我不是单纯做向量检索，而是结合关键词、AST 结构分析和文件读取工具来辅助 Agent 多步决策。”

## 十六、最小练习任务

编写一个脚本，输入一个本地 Python 项目路径：

1. 递归读取 `.py` 文件。
2. 过滤掉 `__pycache__` 等无用目录。
3. 按行数或函数定义切片，并保存 `{"file_path": "...", "start_line": ..., "end_line": ...}` 到元数据中。
4. 用 embedding 模型建索引。
5. 用户提问时检索 top-k，并要求 LLM 回答时必须注明行号和文件路径。
