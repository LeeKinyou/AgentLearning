在前四块中，我们学习了如何调用大模型、注册工具、使用 RAG 召回代码片段以及设计 Agent 状态机循环。然而，这些目前都只是在本地脚本里运行的代码。

今天讲第 5 块：**工程后端化**。它的本质是：**把 LLM、RAG 和 Agent 从“跑在控制台的命令行脚本”转化为“稳定、可观测、可配置、高可用的 Web 服务”。**

对于想找实习或做真实落地项目的开发者来说，这一块是拉开你与普通“Prompt 工程师”差距的关键。

---

## 一、为什么需要工程后端化？

在真实的生产环境中，Agent 不能只运行在个人的电脑终端里。它需要面对以下实际问题：
1. **多用户并发访问**：需要支持会话管理与连接保持。
2. **长耗时请求处理**：Agent 思考和调用工具通常需要十几秒甚至数分钟，HTTP 直连容易超时，需要**异步任务调度**与**事件流推送**。
3. **可观测性与追踪**：模型生成了幻觉或者陷入死循环，必须有结构化日志和分布式追踪（如 Langfuse）记录每一步动作。
4. **安全边界防范**：模型直接生成的工具参数不可信，必须有后端的严格校验拦截，防止文件越权读写或执行恶意命令。
5. **多模型灾备切换**：当主力模型 API 欠费、被限流或网络超时，需要自动 Fallback 到备用模型。

---

## 二、FastAPI 封装：让 Agent 变成 Web 服务

Python 生态中，FastAPI 是构建大模型与 Agent 服务的最主流框架。它支持原生异步（`async/await`）、自动生成交互式 API 文档，且与 Pydantic 数据验证契合度极高。

一个典型的 Agent 服务架构包含：
- **接口路由层 (API Layer)**：负责解析 HTTP 请求，校验参数，快速返回任务 ID。
- **业务服务层 (Service Layer)**：管理任务的创建、状态更新和存储。
- **执行引擎层 (AgentRunner)**：在后台异步执行 Agent 的多步循环逻辑。
- **第三方网关与客户端 (LLMClient & ToolRegistry)**：统一模型调用和工具执行。

---

## 三、Pydantic 请求与响应模型：严格的参数校验

通过定义 Pydantic 数据模型，我们在接口层就能对用户传入的数据做校验，并在自动生成的文档中展示规范的输入输出示例。

```python
from pydantic import BaseModel, Field
from typing import Optional, List

class AnalyzeRequest(BaseModel):
    repo_path: str = Field(..., description="本地或挂载的代码仓库绝对路径")
    question: str = Field(..., description="用户的提问或报错日志内容")
    temperature: Optional[float] = Field(0.2, description="模型随机性参数")
    stream: Optional[bool] = Field(False, description="是否启用流式推送")

class EvidenceItem(BaseModel):
    file_path: str
    start_line: int
    end_line: int
    content: str
    reason: str

class AnalyzeResponse(BaseModel):
    task_id: str = Field(..., description="异步分析任务的唯一标识 ID")
    status: str = Field(..., description="任务当前状态: pending/running/completed/failed")
    answer: Optional[str] = Field(None, description="分析出的最终结论")
    evidence: Optional[List[EvidenceItem]] = Field(None, description="支撑结论的代码证据列表")
```

## 四、统一大模型调用封装：LLMClient 与鲁棒性设计

不要在业务代码中直接调用 `openai.ChatCompletion`。必须统一封装一个 `LLMClient`，处理以下生产级问题：
1. **重试机制 (Retry)**：针对 API 网络抖动，引入**指数退避重试**（如前 3 次重试每次间隔时间翻倍）。
2. **断路器 (Circuit Breaker)**：若服务商 API 持续报错（如触发限流 RateLimit），自动熔断该模型，防止请求积压。
3. **降级逻辑 (Fallback)**：当强模型不可用时，自动切换到性价比更高的小模型（如从 `gpt-4o` 自动降级为本地 `qwen` 实例）。

## 五、工具统一管理：ToolRegistry 机制

我们需要将所有的工具注册到统一的 `ToolRegistry` 中，由其完成：
- 自动根据 Python 函数的 `docstring` 与类型标注生成模型所需的 JSON Schema。
- 根据模型输出的 `tool_name` 动态匹配并安全地调用对应的 Python 函数。
- 校验模型传入的参数是否符合函数签名。

```python
class ToolRegistry:
    def __init__(self):
        self._tools = {}

    def register(self, func):
        # 根据函数签名和注释自动生成 Tool Schema 并存入 self._tools
        self._tools[func.__name__] = {
            "func": func,
            "schema": self._generate_schema(func)
        }
        return func

    def execute(self, tool_name: str, arguments: dict):
        if tool_name not in self._tools:
            raise ValueError(f"未注册的工具: {tool_name}")
        # 参数清洗与安全验证后执行函数
        return self._tools[tool_name]["func"](**arguments)
```

## 六、任务执行引擎与状态管理：AgentRunner

由于 Agent 的执行过程涉及多步 LLM 调用和工具执行，极其耗时，所以我们在后端应当使用**后台异步任务（BackgroundTasks）**或任务队列（如 Celery）来处理请求。

- 用户调用 `POST /api/v1/analyze`，后端立即生成 `task_id`，向 SQLite 中写入一条初始化记录。
- 启动 `BackgroundTasks` 执行 `AgentRunner.run(task_id)`，并在 HTTP 请求中立即将 `task_id` 返回给前端。
- 前端拿到 `task_id` 后，通过**轮询接口**或 **Server-Sent Events (SSE)** 来查看任务状态及执行轨迹。

---

## 七、智能网关与模型路由：LiteLLM 实践

在复杂的 Agent 系统中，我们会使用到多种不同的模型。引入智能网关（如 LiteLLM）可以帮助我们：
- 使用统一的 OpenAI API 规范调用数百种大模型。
- 进行多模型负载均衡和基于权重的路由控制。
- 监控各个 Key 的 Token 消耗与费用开销。

## 八、可观测性与 Trace 记录

Agent 的调试比传统后端困难得多，因为模型推理是黑盒。我们必须在后端记录每一步的 Trace：
- 记录模型每一轮的原始 Prompt 与返回内容。
- 记录工具调用的耗时、入参及返回。
- 记录总的 Token 消耗、费用和执行链路（如集成 Langfuse 平台进行可视化监控）。

## 九、异步事件流推送 (SSE)

对于耗时较长的 Agent 分析任务，直接使用普通的阻塞式 HTTP 请求极易超时。最优雅的设计是使用 **Server-Sent Events (SSE)**，向客户端流式推送 Agent 的思考过程与工具调用事件。

在 FastAPI 中，我们可以定义一个事件流接口：
```python
@app.get("/api/tasks/{task_id}/events")
async def get_task_events(task_id: str):
    async def event_generator():
        # 从队列或 Redis Pub/Sub 中读取该任务执行中产生的事件并 yield 推送
        ...
    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

服务端在执行任务时，会不断向该通道推送结构化事件：

```json
{
  "event": "tool_call",
  "message": "正在搜索相关代码",
  "step": 2
}
```
```json
{
  "event": "evidence_found",
  "message": "找到 pyproject.toml 中的包配置",
  "file_path": "pyproject.toml"
}
```

这样，前端就能实时渲染出一个“正在加载目录”、“正在搜索 `QueryEngine`”、“正在读取 `pyproject.toml`”的动态加载页面，让项目看起来非常成熟和专业。

## 十、Docker：让项目能被别人一键跑起来

很多面试官看项目时，最怕的是：README 写了一堆，但由于依赖版本、本地路径硬编码、系统环境差异等原因跑不起来。
使用 Docker 可以打包你的运行环境，实现“一键启动”。

### 1. 使用 `uv` 的最小 Dockerfile
由于我们项目采用 `uv` 作为包管理工具，在 Docker 中我们可以借助 `uv` 来加速依赖安装并保证环境完全一致：

```dockerfile
FROM python:3.12-slim

# 安装 uv
COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /bin/

WORKDIR /app

# 复制依赖定义文件
COPY pyproject.toml uv.lock ./

# 使用 uv 同步依赖（不创建虚拟环境，直接装到系统 Python 环境中，适合容器）
RUN uv pip install --system -e .

# 复制项目代码
COPY . .

EXPOSE 8000

# 启动 FastAPI 服务
CMD ["uv run uvicorn repomind.api.main:app --host 0.0.0.0 --port 8000"]
```

### 2. 编写 `docker-compose.yml` 组装服务
我们可以将 API 服务与外部存储（如 Redis/SQLite/向量库）进行统一编排：

```yaml
services:
  repomind-api:
    build: .
    ports:
      - "8000:8000"
    env_file:
      - .env
    volumes:
      - ./data:/app/data

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
```

在 README 中，用户只需执行以下命令，即可一键拉起整个 Agent 后端：
```bash
git clone <your-repo>
cd AgentLearning
cp .env.example .env
docker compose up --build
```

## 十二、数据存储：任务、日志与索引都要落地

不要让所有的 Agent 执行状态只存在于内存中，一旦重启服务数据就会丢失。最小版本可以用 SQLite 进行持久化存储。

你可以设计两张核心表：

### 1. `tasks` 表
- `task_id` (主键)
- `status` (pending/running/completed/failed)
- `question` (原始输入)
- `repo_path` (被分析的仓库)
- `result` (最终结论)
- `error` (报错原因)
- `created_at` / `updated_at`

### 2. `agent_steps` 表
- `id` (主键)
- `task_id` (外键)
- `step_index` (第几步)
- `step_type` (thought/tool_call/observation)
- `tool_name` (调用工具名)
- `input` (工具入参)
- `output` (工具执行结果)
- `latency_ms` (该步延迟)

通过这两张表，你的后端就能提供完整的任务执行历史查询，并可以还原出 Agent 执行的完整轨迹图。

## 十三、安全控制：Agent 不能随便碰系统

模型是由 Prompt 驱动的，极易遭受 Prompt 注入攻击。如果模型试图越权操作，后端必须进行“防御性编程”。

- **限制访问目录**：防止模型通过传入 `../../etc/passwd` 读取容器外的敏感文件。每次读取文件前，后端必须做路径解析和校验：
  ```python
  from pathlib import Path

  def validate_path(repo_root: str, file_path: str) -> Path:
      abs_root = Path(repo_root).resolve()
      abs_file = (abs_root / file_path).resolve()

      # 判定绝对路径是否以 repo_root 开头，防范目录穿越漏洞
      if not str(abs_file).startswith(str(abs_root)):
          raise PermissionError("Access denied: 不能访问项目根目录之外的文件。")

      return abs_file
  ```
- **写入操作限制**：涉及写文件（`write_file`）或运行命令（`run_command`）这类高危工具，默认设计为挂起状态，在 SSE 推送中发出确认请求，等待用户在前端点击“确认”后后端才继续执行。
- **敏感信息脱敏**：记录日志时，必须过滤或脱敏 API Key、用户密码等机密敏感数据。

## 十四、工程后端化在 RepoMind 里怎么落地？

建议按优先级逐步实现：

### P0 级（必须要做的核心闭环）
1. 使用 **FastAPI** 封装分析和状态查询接口。
2. 封装 `LLMClient` 统一处理模型重试和参数控制。
3. 建立 `ToolRegistry`，实现参数自动校验。
4. 编写 `AgentRunner` 管理多步 while loop 推理。
5. 编写结构化日志记录和 `.env` 环境变量配置。

### P1 级（体现明显工程感的加分项）
1. 设计任务表与步骤表，使用 **SQLite** 存储任务历史。
2. 使用 **SSE (Server-Sent Events)** 实现步骤执行轨迹的实时推送。
3. 编写 `Dockerfile`，支持使用 `uv` 缓存一键构建镜像。
4. 完善 API 文档中的示例，提供 `.env.example`。

### P2 级（选做，锦上添花）
1. 引入 Redis，实现多任务并发情况下的异步队列调度（Celery / RQ）。
2. 在工具链中增加沙箱机制（如在 Docker 容器或临时虚拟环境中运行 `pytest`）。
3. 设计 Web 端 UI 仪表盘，展示总的 Token 消耗和调用成本。

> [!TIP]
> **面试回答话术**：
> “我的项目不是简单脚本，而是把 Agent 能力封装成了 FastAPI 服务。接口层只负责请求校验和响应，核心逻辑放在 Service 和 AgentRunner 中。AgentRunner 维护任务状态和执行步骤，ToolRegistry 负责统一注册和调用工具，LLMClient 负责模型调用、重试和结构化输出解析。整个过程会记录 task_id、工具调用、模型耗时、token 消耗和错误日志，方便调试和追踪。”

当面试官追问：
- **“Agent 运行过慢，导致 HTTP 连接超时怎么处理？”**
  - 回答：“我将分析任务设计为异步后台任务。用户提交分析请求后，API 层会立刻生成并返回任务 ID。随后后台启动 `AgentRunner` 执行决策循环。用户端通过 SSE 建立连接，实时监听执行步骤的推送。这既避免了连接超时，又提供了良好的用户可观测性。”
- **“如何保证大模型调用工具的系统安全？”**
  - 回答：“后端坚决不给模型提供直接的系统操作权限。所有的工具调用必须经过 `ToolRegistry` 进行入参校验与白名单限制。例如读取文件工具，后端会对路径进行 `resolve()` 解析，强制验证其前缀是否属于项目根目录，彻底防范目录穿越漏洞。同时，对于高风险的写入和执行操作，会进入任务挂起状态，要求用户手动点击确认后才能在后端执行。”
